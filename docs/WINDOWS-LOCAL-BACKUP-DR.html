1. Backup principles

Objective: After hardware failure, corruption, or accidental deletion, restore clinical operations within hours using offline backups — independent of Railway DR (which is covered in the replication guide).
LayerWhat to protectPriority
Databasehms_demo (all tbl_* data)Critical — nightly minimum
Configuration.env, config/, uploaded assetsCritical
ApplicationHMS_JS deploy folder or build packageHigh — weekly snapshot
SystemWindows image (optional)Medium — monthly

2. Backup layout & 3-2-1 rule

D:\HMS-Backups\ ├── mysql\ │ ├── daily\ ← mysqldump .sql.gz (retain 14 days) │ └── hourly\ ← optional (retain 48 hours) ├── config\ │ └── daily\ ← .env, betterpay.json, brand files ├── app\ │ └── weekly\ ← zip of deploy folder (no node_modules) ├── logs\ │ └── backup-*.log └── MANIFEST.txt ← last successful backup summary 3-2-1 rule: 3 copies · 2 media types · 1 off-site (USB + cloud sync)
  • Copy 1: Live production on C:
  • Copy 2: D:\HMS-Backups on separate disk/partition
  • Copy 3: External USB or encrypted cloud (OneDrive/Google Drive sync of mysql/daily only)

3. MySQL database backup

3.1 Manual dump (verification)

"C:\xampp\mysql\bin\mysqldump.exe" -h127.0.0.1 -uroot -p ^
  --single-transaction --routines --triggers --events ^
  hms_demo > D:\HMS-Backups\mysql\manual-test.sql

3.2 Compressed daily backup

The automated script gzips dumps to save space. Verify periodically:

"C:\xampp\mysql\bin\mysql.exe" -uroot -p -e "SELECT COUNT(*) FROM hms_demo.tbl_patient"

3.3 MySQL data directory (optional)

For advanced recovery, stop MySQL and copy C:\xampp\mysql\data\hms_demo — only when MySQL is stopped. Logical dumps are safer for routine backups.

Never copy live InnoDB files while MySQL is running — risk of corrupt backup.

5. Automated backup script

Run from repository:

powershell -ExecutionPolicy Bypass -File C:\HMS_JS\scripts\windows-hms-backup.ps1

The script:

  • Creates dated MySQL dump under D:\HMS-Backups\mysql\daily\
  • Copies .env and config\ to config\daily\
  • Prunes dumps older than 14 days
  • Writes log to D:\HMS-Backups\logs\

5.1 Optional parameters

powershell -File scripts\windows-hms-backup.ps1 `
  -BackupRoot "D:\HMS-Backups" `
  -DbName "hms_demo" `
  -DbUser "root" `
  -DbPassword "your-password" `
  -XamppBin "C:\xampp\mysql\bin"

  2. Trigger: daily 02:00, wake computer if needed.
  3. Action: powershell.exe -ExecutionPolicy Bypass -File C:\HMS_JS\scripts\windows-hms-backup.ps1
  4. Conditions: start only if AC power (for laptops).
  5. Settings: restart on failure up to 3 times.

7. Full restore procedure (server crash)

Perform restores on a test path first if time permits. In emergency, follow steps sequentially.
  1. Prepare hardware — Windows 10, install XAMPP, Node.js 18+, Git or copy deploy folder.
  2. Start MySQL in XAMPP Control Panel.
  3. Create database & user (if fresh XAMPP): 
    CREATE DATABASE hms_demo CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'root'@'localhost' IDENTIFIED BY 'your-password';
GRANT ALL PRIVILEGES ON hms_demo.* TO 'root'@'localhost';
FLUSH PRIVILEGES;
  4. Restore database from latest backup: 
    powershell -ExecutionPolicy Bypass -File C:\HMS_JS\scripts\windows-hms-restore.ps1 `
  -SqlBackup "D:\HMS-Backups\mysql\daily\hms-YYYYMMDD-HHmmss.sql.gz"
  5. Restore application — copy HMS_JS from weekly zip or Git; run npm install.
  6. Restore config — copy .env from D:\HMS-Backups\config\daily\.
  7. Start HMS — npm start or Windows service (Section 9).
  8. Verify — http://localhost:3000/__health, login, patient count, print test.
  9. Re-enable backups and Railway sync tasks.

8. Partial recovery scenarios

ScenarioAction
Corrupt single tableRestore table from dump: mysql ... < extract_table.sql
Deleted .env onlyRestore from config\daily\
Bad deploy / code bugRevert to last weekly app zip; DB unchanged
RansomwareDo not pay — rebuild from offline USB backup (not synced folder)
Disk fullFree space; prune old backups; expand partition

9. Windows service & auto-start

Run HMS as a service so it survives logout and reboots (see main Local Deployment guide Section 19). After restore:

  • Configure XAMPP MySQL as Windows service (auto start)
  • Configure HMS Node service with restart on failure
  • Set service account with read access to .env only — not Domain Users

10. Testing & runbook

  • Monthly: restore latest dump to hms_demo_test on same server
  • Quarterly: full restore drill on spare PC or VM
  • Document backup size trend (alert if sudden 50%+ drop)
  • Store printed runbook with Railway DR URL and IT contacts
  • Verify USB off-site copy is unplugged after sync (ransomware protection)
  • After any restore, run node diagnostic2.js
