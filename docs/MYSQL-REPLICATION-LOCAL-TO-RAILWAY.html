1. Strategy & terminology

Goal: If the Windows 10 hospital server fails (hardware, power, disk, OS), staff can continue on the Railway-hosted HMS using data synced from the local MySQL database.
TermMeaning for this guide
PrimaryLocal Windows HMS + local MySQL (daily production)
Secondary / DRRailway HMS + Railway MySQL (warm standby)
RPORecovery Point Objective — max data loss = time since last sync
RTORecovery Time Objective — time to switch users to Railway URL
Logical syncmysqldump → import (recommended, included script)
Native replicationMySQL binlog streaming (advanced; Section 6)
Important: This is warm standby sync, not live active-active. Only one site should accept writes at a time to avoid split-brain conflicts.

3. Railway MySQL setup

  1. In your Railway project, click New → Database → MySQL.
  2. Open the MySQL service → Variables tab. Note:
    • MYSQLHOST (public proxy, e.g. crossover.proxy.rlwy.net)
    • MYSQLPORT
    • MYSQLUSER / MYSQLPASSWORD
    • MYSQLDATABASE (often railway)
  3. Ensure the MySQL plugin allows public networking (required for sync from hospital WAN).
  4. Restrict access: use strong password; consider IP allowlisting if Railway plan supports it.

4. Railway HMS app (standby)

Deploy the main HMS application as a second Railway service in the same project (or separate project linked to same MySQL).

SESSION_SECRETSame as local OR different (users re-login on failover)

Generate Railway HTTPS domain for the HMS service. Keep this URL documented for failover.

During normal operations, train staff to use the local URL. Railway is the emergency URL only (or use DNS failover — Section 7).

5. Scheduled logical sync (recommended)

The repository includes scripts/sync-mysql-to-railway.js which:

  1. Dumps local MySQL with mysqldump (XAMPP)
  2. Connects to Railway MySQL over TLS TCP
  3. Drops and re-imports tables (full refresh)
  4. Verifies table count and sample row counts

5.1 Configure sync environment file

Create C:\HMS_JS\docs\scripts\railway-sync.env (do not commit secrets):

# Local XAMPP MySQL (primary)
LOCAL_DB_HOST=localhost
LOCAL_DB_PORT=3306
LOCAL_DB_USER=root
LOCAL_DB_PASSWORD=your-local-password
LOCAL_DB_NAME=hms_demo
XAMPP_MYSQL_BIN=C:\xampp\mysql\bin

# Railway MySQL (from plugin variables)
RAILWAY_MYSQL_HOST=crossover.proxy.rlwy.net
RAILWAY_MYSQL_PORT=36338
RAILWAY_MYSQL_USER=root
RAILWAY_MYSQL_PASSWORD=your-railway-password
RAILWAY_MYSQL_DATABASE=railway

# Optional dump path
SYNC_DUMP_PATH=C:\HMS-Backups\railway-sync\latest.sql

5.2 Manual test run

cd C:\HMS_JS
set RAILWAY_MYSQL_HOST=crossover.proxy.rlwy.net
set RAILWAY_MYSQL_PORT=36338
set RAILWAY_MYSQL_USER=root
set RAILWAY_MYSQL_PASSWORD=your-password
set RAILWAY_MYSQL_DATABASE=railway
node scripts/sync-mysql-to-railway.js

5.3 Automate with Windows Task Scheduler

  1. Create folder C:\HMS-Backups\railway-sync\logs.
  2. Create C:\HMS_JS\docs\scripts\run-railway-sync.ps1: 
    $envFile = "C:\HMS_JS\docs\scripts\railway-sync.env"
Get-Content $envFile | ForEach-Object {
  if ($_ -match '^([^#=]+)=(.*)$') { Set-Item -Path "Env:$($matches[1].Trim())" -Value $matches[2].Trim() }
}
cd C:\HMS_JS
node scripts/sync-mysql-to-railway.js *>&1 | Tee-Object -FilePath "C:\HMS-Backups\railway-sync\logs\sync-$(Get-Date -Format yyyyMMdd-HHmm).log"
  3. Task Scheduler → Create Task:
    • Trigger: every 15 minutes (or 30/60 based on RPO needs)
    • Action: powershell.exe -ExecutionPolicy Bypass -File C:\HMS_JS\docs\scripts\run-railway-sync.ps1
    • Run whether user is logged on or not · highest privileges
    • Run only when network is available
  4. Schedule during low activity if full dump is large; test duration first.

5.4 Sync window & production impact

  • mysqldump --single-transaction avoids long table locks on InnoDB tables.
  • Large databases (multi-GB) may take several minutes — adjust interval accordingly.
  • During import, Railway DB is briefly inconsistent — run HMS on Railway in maintenance mode or keep it unused until sync completes.

6. Native MySQL replication (advanced)

For near-real-time RPO, MySQL binary log replication can stream changes from local master to Railway replica. This is more complex and may be limited by Railway MySQL privileges.

6.1 When to use

  • Database > 2 GB and full dump every 15 min is too slow
  • RPO requirement < 5 minutes
  • IT staff comfortable with MySQL replication administration

6.2 Local master (XAMPP my.ini)

[mysqld]
server-id=1
log-bin=mysql-bin
binlog_format=ROW
expire_logs_days=7
bind-address=127.0.0.1
Opening MySQL to the public internet for replication is a security risk. Prefer VPN or Railway private networking if available. Never expose XAMPP MySQL to 0.0.0.0 without firewall IP restrictions.

6.3 Railway replica

Create replication user on master, take initial snapshot, configure Railway MySQL as replica with CHANGE REPLICATION SOURCE TO (MySQL 8). Verify SHOW REPLICA STATUS\G — Replica_IO_Running: Yes.

If Railway managed MySQL blocks replication commands, remain on Section 5 logical sync.

7. Failover procedure (local → Railway)

Stop writes on primary first. If primary is already dead, skip to step 3.
  1. Confirm primary failure — server unreachable, MySQL down, or unrecoverable within RTO.
  2. Run final sync if primary MySQL is still readable (from another machine or disk mount): 
    node scripts/sync-mysql-to-railway.js
  3. Verify Railway database — connect with MySQL client; check tbl_patient row count vs last known good.
  4. Start / verify Railway HMS — open /__health on Railway URL.
  5. Redirect users:
    • Update DNS A/CNAME for hospital hostname to Railway, OR
    • Communicate emergency URL to all departments, OR
    • Update reverse proxy / firewall rules
  6. Update license server — ensure Railway HMS HMS_PUBLIC_URL matches active URL for push webhooks.
  7. Post incident — document data loss window (time since last successful sync).

8. Fail-back to local Windows server

  1. Repair or replace local hardware; restore OS and XAMPP from backup (see Backup DR guide).
  2. Option A: Dump Railway MySQL → import to local (reverse sync — adapt script swapping LOCAL/RAILWAY).
  3. Option B: Restore local from latest local backup then replay changes manually if gap is small.
  4. Run full UAT on local before switching users back.
  5. Re-enable scheduled sync local → Railway once local is primary again.
