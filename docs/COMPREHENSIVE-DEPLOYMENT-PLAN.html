1. Executive overview

Goal: Safely promote tested HMS application code from the development workspace to a hardened external Windows server, while maintaining a warm copy of the hospital database on Railway for disaster recovery and optional cloud workflows.

This plan covers three coordinated workstreams:

WorkstreamScopePrimary artifacts
Application deployDev build → Update folder → production Program Files installbuild-update-package.ps1, apply-deploy-update.ps1
Database migrationLocal MySQL (XAMPP / Windows) → Railway MySQLsync-mysql-to-railway.js, bidirectional-sync-local-railway.js
OperationsBackups, service install, NTFS hardening, restoredocs/scripts/04–12, nightly backup task
Single writer rule: Only one environment should accept clinical writes at a time. Scheduled sync is warm standby (local → Railway), not active-active. Use bidirectional sync only with explicit table scope and change-control.

3. Pre-deployment checklist

Phase 0 — Before any production change
  • All changes tested on dev: login, OPD queue, billing, pharmacy, statutory reports
  • Route audit clean: node scripts/audit-routes.js
  • Smoke routes pass: node scripts/smoke-routes.js
  • UI rebuilt: npm run build:ui produces fresh hms-ui.js (~1.5–2.2 MB)
  • Production backup verified (DB dump + file backup within 24 h)
  • Deploy window communicated to clinical staff
  • Administrator PowerShell available on deploy server
  • Update folder copied and verified (see Section 5)
  • Railway sync env configured if DB migration is in scope
  • License keys and LICENSE_SERVER_URL unchanged unless planned

4. Build update package (development machine)

Phase 1 — Dev workspace

Run on C:\HMS_JS after merging and testing all changes:

powershell -ExecutionPolicy Bypass -File C:\HMS_JS\scripts\build-update-package.ps1

This script:

  1. Runs npm run build:ui (React bundles: hms-ui.js, hms-login.js, hms-print.js)
  2. Runs node scripts/build-deploy-package.js → dist\hms-deploy
  3. Robocopies deploy output to C:\HMS_JS\Update (replaces previous Update folder)
  4. Writes UPDATE-README.txt with timestamp and verify steps
Critical: Deploying backend files without public\dist\hms-ui.js leaves users on an old UI. Always confirm bundle size and date before copying to production.

4.1 Optional dev data export

For catalog/pharmacy seed on production (only when intentional):

node scripts/export-pharmacy-deploy-data.js
# Produces Update\data\deploy-data-export.json

Apply on server with -ImportDeployData flag (Section 6).

5. Transfer to deploy server

Phase 2 — Physical or remote copy

Copy the entire C:\HMS_JS\Update folder to the deploy server (same path recommended):

  • USB drive, RDP drag-and-drop, or network share
  • Include scripts\apply-deploy-update.ps1, all routes, lib, views, public\dist\

5.1 Verify after copy

  • C:\HMS_JS\Update\app.js exists
  • C:\HMS_JS\Update\public\dist\hms-ui.js exists (~1500–2200 KB)
  • C:\HMS_JS\Update\scripts\apply-deploy-update.ps1 present
  • New modules present (e.g. routes\statutoryReports.js, lib\erOrderAutoCharge.js)
  • MANUAL-COPY-README.txt reviewed
Never overwrite on deploy server: production .env, uploads\, tmp\, patient attachments, or DIPE export data.

6. Apply production update

Phase 3 — Deploy server (Administrator)

6.1 Automated apply (recommended)

powershell -ExecutionPolicy Bypass -File C:\HMS_JS\Update\scripts\apply-deploy-update.ps1

The script performs:

  1. Stop HMS — NSSM service ZAIZENS-HMS or running node.exe processes
  2. Backup — robocopy to deploy-backups\HMS-backup-YYYYMMDD-HHmm (with /B for backup mode on hardened trees)
  3. ACL unlock — grants Administrators modify on install tree (required after NTFS hardening)
  4. Apply files — robocopy Update → C:\Program Files\ZAIZENS\HMS, excluding .env
  5. Verify UI bundle — confirms public\dist\hms-ui.js present
  6. Optional data import — -ImportDeployData or -PullPharmacyFromRailway
  7. Start HMS — NSSM start or background node app.js

6.2 Script flags

FlagEffect
-SkipBackupSkip pre-deploy robocopy backup (use only if recent backup exists)
-SkipServiceRestartDo not stop/start service (manual control)
-ImportDeployDataImport dev catalog export from Update\data\
-PullPharmacyFromRailwayPull pharmacy tables from Railway to local
-HmsRootOverride install path (default: Program Files\ZAIZENS\HMS)
-UpdateSourceOverride Update folder path

6.3 Manual robocopy (fallback)

nssm stop ZAIZENS-HMS
robocopy "C:\Program Files\ZAIZENS\HMS" "D:\HMS-Data\deploy-backups\HMS-backup-YYYYMMDD" /E /B
robocopy C:\HMS_JS\Update "C:\Program Files\ZAIZENS\HMS" /E /B /IS /IT /XF .env
nssm start ZAIZENS-HMS

7. Local MySQL → Railway migration

Phase 4 — Database replication

Warm standby sync keeps Railway MySQL aligned with the on-premises primary for disaster recovery. This is not a one-time cutover unless you explicitly fail over (Section 11).

7.1 Strategy summary

ModeScriptUse case
One-way full refreshscripts/sync-mysql-to-railway.jsInitial Railway seed, scheduled DR sync
Table-scoped syncdocs/scripts/sync-local-railway.jsDirectional sync of named tables
Replicate (local → Railway)docs/scripts/replicate-local-to-railway.jsIncremental table replication
Bidirectionaldocs/scripts/bidirectional-sync-local-railway.jsControlled two-way sync (advanced)

7.2 Railway MySQL setup

  1. Railway project → New → Database → MySQL
  2. Note variables: MYSQLHOST, MYSQLPORT, MYSQLUSER, MYSQLPASSWORD, MYSQLDATABASE
  3. Enable public networking on MySQL plugin (required for sync from hospital WAN)
  4. Use strong passwords; restrict source IP if Railway plan supports allowlisting

7.3 Configure sync environment

Copy example and edit (do not commit secrets):

copy docs\scripts\railway-sync.env.example docs\scripts\railway-sync.env
# Local primary (XAMPP or Windows MySQL)
LOCAL_DB_HOST=localhost
LOCAL_DB_PORT=3306
LOCAL_DB_USER=root
LOCAL_DB_PASSWORD=***
LOCAL_DB_NAME=hms_demo
XAMPP_MYSQL_BIN=C:\xampp\mysql\bin

# Railway MySQL (from plugin)
RAILWAY_MYSQL_HOST=crossover.proxy.rlwy.net
RAILWAY_MYSQL_PORT=36338
RAILWAY_MYSQL_USER=root
RAILWAY_MYSQL_PASSWORD=***
RAILWAY_MYSQL_DATABASE=railway

SYNC_DUMP_PATH=C:\HMS-Backups\railway-sync\latest.sql

Alternatively, place db-sync.env in repo root or scripts\. The sync core loads HMS .env DB_* when LOCAL_DB_USE_HMS_ENV is not disabled.

7.4 Initial migration (full dump)

cd C:\HMS_JS
powershell -ExecutionPolicy Bypass -File docs\scripts\run-railway-sync.ps1

Or directly:

node scripts/sync-mysql-to-railway.js

Process:

  1. mysqldump --single-transaction from local MySQL
  2. Connect to Railway MySQL over TLS
  3. Drop and re-import tables (full refresh)
  4. Verify table counts and sample row checks
Large databases (multi-GB) may take several minutes. Schedule during low activity. Railway HMS should remain in maintenance or unused until sync completes.

8. Bidirectional & scheduled sync

8.1 Scheduled one-way sync (recommended DR)

Register Windows Task Scheduler to run every 15–60 minutes:

powershell -ExecutionPolicy Bypass -File docs\scripts\register-replicate-scheduled-task.ps1

Logs: C:\HMS-Backups\railway-sync\logs\ or tmp\railway-sync-logs\

8.2 Bidirectional sync (controlled)

powershell -ExecutionPolicy Bypass -File scripts\run-bidirectional-sync.ps1
powershell -ExecutionPolicy Bypass -File scripts\run-bidirectional-sync.ps1 -DryRun
powershell -ExecutionPolicy Bypass -File scripts\run-bidirectional-sync.ps1 -Tables tbl_service_catalog,tbl_inventory_item
Split-brain risk: Bidirectional sync can overwrite production data. Use table filters, dry-run first, and change-control. Normal operations: local primary → Railway only.

10. Rollback procedures

10.1 Application rollback

  1. Stop HMS service
  2. Restore from deploy backup: deploy-backups\HMS-backup-*
    3. robocopy "D:\HMS-Data\deploy-backups\HMS-backup-YYYYMMDD-HHmm" "C:\Program Files\ZAIZENS\HMS" /E /B /IS /IT
  3. Confirm production .env intact
  4. Start service; hard refresh clients

10.2 Database rollback

powershell -ExecutionPolicy Bypass -File docs\scripts\11-restore-database.ps1 -DumpFile "D:\HMS-Data\mysql-backups\hms_demo_YYYYMMDD.sql"

Requires HMS stopped; restores from nightly mysqldump. Document any data entered after backup time as lost.

11. Disaster recovery & failover

Stop writes on primary first. If primary is dead, run final sync from readable backup media if possible, then activate Railway.
  1. Confirm primary failure (hardware, MySQL, network)
  2. Final sync if local MySQL readable: node scripts/sync-mysql-to-railway.js
  3. Verify Railway DB — patient row counts vs last known good
  4. Start Railway HMS; test /__health
  5. Redirect users: DNS, emergency URL broadcast, or proxy rules
  6. Update HMS_PUBLIC_URL on license server for push webhooks
  7. Document data loss window (time since last sync)

11.1 Fail-back to local

  1. Repair local server; restore OS and MySQL from backup
  2. Reverse sync: dump Railway → import local (swap LOCAL/RAILWAY in env), or restore local backup + manual gap fill
  3. Full UAT on local before switching users back
  4. Re-enable scheduled local → Railway sync

12. Security & NTFS hardening

Production installs under Program Files use restricted ACLs (06-apply-ntfs-hardening.ps1). Deploy scripts temporarily grant Administrators modify access during updates.

  • Run deploy as Administrator — non-elevated runs produce ACL noise and partial copies
  • Never deploy private license signing keys to hospital servers
  • Keep railway-sync.env and MySQL credentials out of git
  • Do not expose XAMPP MySQL to 0.0.0.0 without firewall IP restrictions
  • Rotate MySQL passwords via 01b-rotate-to-hardened-password.sql after initial install
