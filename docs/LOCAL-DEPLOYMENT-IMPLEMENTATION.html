1. Executive summary

This document is the authoritative implementation guide for deploying ZAIZENS Integrated HMS at Franciscan ZAIZENS Integrated HMS — ZAIZENS SOA, abbreviated ZAIZENS. The system runs on a dedicated Windows server inside the hospital network and serves clinical, operational, HR, and financial workflows through role-based portals.

The guide covers full installation (Node.js, MySQL, application build, Windows service), TSSF-specific branding, operational validation, and a defence-in-depth security model that addresses two distinct risks:

  • Unauthorized modification of application source, configuration, or licensing logic on the server.
  • Unauthorized access to the server, database, admin functions, or patient data by staff or outsiders.
Important distinction: HMS is a Node.js application. Unlike compiled desktop software, some runtime files (EJS views, server JavaScript) must remain on disk in a form Node can execute. Protection therefore combines minimal deployment artifacts, operating-system hardening, access control, and optional obfuscation — not a single “hide all code” switch.

7. Pre-deployment checklist

  • Windows Server joined to hospital domain (recommended) or standalone with local admin policy
  • Server hostname documented (e.g. TSSF-HMS-01)
  • Static IP assigned on server VLAN
  • Node.js 18+ installed (64-bit), verified with node -v
  • MySQL 8 installed as Windows service, root password vaulted
  • Service account svc-hms created — no interactive logon, password vaulted
  • Antivirus exclusion configured for uploads/ scan-on-write (not blind exclude of entire app)
  • Backup destination ready (NAS or secondary disk) — encrypted
  • Deployment package built with npm run build:deploy:protected
  • TSSF .env prepared offline — not emailed in plain text
  • SSL certificate obtained for internal DNS name (e.g. hms.tssf-soa.local)
  • Implementation team has Super Admin credentials for first login
  • Change control register opened for go-live window

  2. Create installation paths 
    mkdir "C:\Program Files\ZAIZENS\HMS"

mkdir "D:\HMS-Data\mysql-backups"
mkdir "D:\HMS-Data\hms-backups"

  7. Configure MySQL — see Section 9.

15. Protecting source code from modification

Attackers or curious staff with write access to the HMS folder could alter app.js, disable licensing checks, expose hidden routes, or inject backdoors. The following controls are mandatory for ZAIZENS production.

15.1 Principle: immutable application, mutable data only

PathPermissions (svc-hms)Permissions (Administrators)Permissions (Users / staff)
C:\Program Files\ZAIZENS\HMS\ (code)Read & executeRead & execute (not Full Control in steady state)No access
uploads\ModifyReadNo access
tmp\ModifyReadNo access
.envReadRead (change via change control only)No access
node_modules\Read & executeRead & executeNo access

15.2 Apply NTFS ACLs (PowerShell — run as Administrator)

$root = "C:\Program Files\ZAIZENS\HMS"
$svc  = "TSSF\svc-hms"   # adjust domain\account

# Reset and inherit off for root
icacls $root /inheritance:r
icacls $root /grant:r "Administrators:(OI)(CI)(RX)"
icacls $root /grant:r "SYSTEM:(OI)(CI)(F)"
icacls $root /grant:r "${svc}:(OI)(CI)(RX)"
icacls $root /deny "Users:(OI)(CI)(F)"

# Writable folders only
icacls "$root\uploads" /grant:r "${svc}:(OI)(CI)(M)"
icacls "$root\tmp"     /grant:r "${svc}:(OI)(CI)(M)"

# Lock .env
icacls "$root\.env" /inheritance:r
icacls "$root\.env" /grant:r "Administrators:(R)"
icacls "$root\.env" /grant:r "SYSTEM:(F)"
icacls "$root\.env" /grant:r "${svc}:(R)"

15.3 Deploy minimal artifacts (no development tree)

Never deploy the Git repository to ZAIZENS. The protected package excludes:

  • .git/ — full source history
  • frontend/src/ — readable React source
  • license-generator/ — vendor signing tool
  • scripts/, docs/, migrations/ — internal tooling
  • Development dependencies — use npm ci --omit=dev

15.4 Optional: JavaScript obfuscation (included in protected build)

npm run build:deploy:protected

Obfuscates lib/ and routes/ using javascript-obfuscator. This deters casual reading and tampering but is not cryptographic security. Combine with NTFS read-only code and change control.

15.5 Optional: bytecode compilation (advanced)

For highest deterrence on server logic, selected modules may be compiled to V8 bytecode (.jsc) using bytenode. Requires a controlled build pipeline — contact ZAIZENS if ZAIZENS contracts bytecode delivery.

15.6 File integrity monitoring

  1. After golden deploy, generate SHA-256 hashes of all files under lib/, routes/, app.js, views/:
Get-ChildItem -Recurse "C:\Program Files\ZAIZENS\HMS\lib","C:\Program Files\ZAIZENS\HMS\routes","C:\Program Files\ZAIZENS\HMS\app.js" |
  Get-FileHash -Algorithm SHA256 |
  Export-Csv "D:\HMS-Data\hms-backups\integrity-baseline.csv"
  1. Schedule weekly hash comparison; alert IT if any application file changed outside an approved patch window.
  2. Enable Windows audit policy for object access on the HMS folder (optional, high volume).

15.7 Change control policy

Production rule: No manual edits to files under C:\Program Files\ZAIZENS\HMS on the live server. All updates must come from a new signed deployment package from ZAIZENS, applied during a documented maintenance window, with pre/post integrity hash verification and rollback plan.
  • Maintain a change register: date, version, approver, files changed, test evidence
  • Take MySQL backup immediately before any upgrade
  • Re-apply NTFS ACLs after upgrade in case installer resets permissions

15.8 What cannot be fully hidden

EJS templates in views/ remain text files Node must read. Minification of EJS is not standard. Protection relies on preventing write access and preventing read access by unauthorised OS users — not on hiding template content from a determined administrator with full server access.

16. Preventing unauthorized access

16.1 Physical and facility security

  • HMS server in a locked server room or locked rack — not under a reception desk
  • USB storage disabled via Group Policy for non-admin accounts (prevents copy-out of .env and DB dumps)
  • BIOS/boot password; disable boot from USB
  • UPS with graceful shutdown script for MySQL and HMS service

16.2 Network access control

  • Place HMS on a dedicated server VLAN; no direct routing to guest Wi-Fi
  • Windows Firewall: inbound allow 443 from workstation subnet only; deny 3000 and 3306 from LAN
  • MySQL bound to 127.0.0.1 only
  • Disable unused Windows features (IIS FTP, Telnet, etc.)
  • Remote support via VPN only — not port-forwarded RDP from internet

16.3 HTTPS and reverse proxy (IIS example)

Do not expose raw Node on port 3000 to the hospital LAN. Terminate TLS at IIS:

  1. Install IIS + URL Rewrite + Application Request Routing (ARR)
  2. Create site hms.tssf-soa.local with TLS certificate
  3. Reverse proxy to http://127.0.0.1:3000
  4. Optional: require hospital domain authentication at IIS layer (double gate)

16.4 Windows server hardening

16.5 Service account isolation

The HMS Windows service must run as svc-hms, not Local System or an admin user:

  • Deny logon locally and deny RDP for svc-hms
  • Grant "Log on as a service" only
  • If service account is compromised, attacker still cannot modify code (read-only ACL)

16.6 Database credential protection

  • Database password Ts$fSOA2026!Hms (15 chars) — in all docs\scripts\ files
  • Credentials only in .env — never in EJS, JavaScript, or Git
  • No phpMyAdmin / MySQL Workbench on server open to LAN
  • Encrypt backup files at rest (BitLocker on D: or backup tool encryption)

Endpoints such as /__health, /__boot expose environment hints. In hardened production, restrict these to localhost or IT subnet via IIS URL Rewrite rules or firewall — not open to all hospital workstations.

18.8 Monthly archive job (optional second task)

On the 1st of each month at 03:00, copy the latest dump to a monthly folder and compress:

$src = Get-ChildItem "D:\HMS-Data\mysql-backups\hms_demo_*.sql" | Sort-Object LastWriteTime -Descending | Select-Object -First 1
$dest = "D:\HMS-Data\mysql-backups\monthly\hms_demo_$(Get-Date -Format yyyy-MM).sql"
Copy-Item $src.FullName $dest -Force
Compress-Archive -Path $dest -DestinationPath "$dest.zip" -Force

Retain 12 monthly archives; purge older files manually or via scheduled script.

18.10 Restore procedure — database

Use case: Database corruption, accidental mass delete, failed upgrade rollback.

  1. Stop HMS service — nssm stop ZAIZENS-HMS (prevents writes during restore).
  2. Identify backup file — e.g. D:\HMS-Data\mysql-backups\hms_demo_2026-05-25_020015.sql.
  3. Optional — restore to test DB first on UAT VM before touching production.
  4. Restore production database:
mysql -u root -p -e "DROP DATABASE IF EXISTS hms_demo; CREATE DATABASE hms_demo CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
mysql -u root -p hms_demo < "D:\HMS-Data\mysql-backups\hms_demo_2026-05-25_020015.sql"
  1. Start HMS service — nssm start ZAIZENS-HMS.
  2. Verify — login, patient search, today's OPD queue, one financial report.
  3. Document incident — record data loss window (RPO achieved) and root cause.

18.11 Restore procedure — uploads folder

nssm stop ZAIZENS-HMS
robocopy "D:\HMS-Data\uploads-backups\2026-05-25_020015" "D:\HMS-Data\uploads" /MIR
nssm start ZAIZENS-HMS

/MIR replaces the live folder to match the backup snapshot. Confirm timestamp folder matches the database restore date for consistency.

18.12 Restore procedure — configuration (.env)

copy "D:\HMS-Data\hms-backups\config\env_2026-05-25_020015.env" "C:\Program Files\ZAIZENS\HMS\.env"
icacls "C:\Program Files\ZAIZENS\HMS\.env" /inheritance:r
icacls "C:\Program Files\ZAIZENS\HMS\.env" /grant:r "Administrators:(R)" "SYSTEM:(F)" "TSSF\svc-hms:(R)"

18.13 Full server disaster recovery

If the HMS server is lost entirely (hardware failure, ransomware):

  1. Provision replacement Windows Server; install Node.js 18+, MySQL 8.
  2. Restore latest vendor deployment package to C:\Program Files\ZAIZENS\HMS\.
  3. Restore .env from config vault; re-apply NTFS ACLs (Section 15).
  4. Restore MySQL database from latest offsite dump (Section 18.10).
  5. Restore uploads/ from offsite mirror (Section 18.11).
  6. Register Windows service and IIS HTTPS (Sections 16, 19).
  7. Run UAT checklist (Section 12) before reopening to staff.
